BROKER SERVICE IMPLEMENTATION ANALYSIS
=======================================
Date: 2026-03-30

EXISTING DHAN INTEGRATIONS (Already built in Python)
=====================================================

1. dhan_option_chain_fetcher.py:
   - Auth: Hardcoded CLIENT_ID + ACCESS_TOKEN
   - Scrip Master: Downloads CSV from https://images.dhan.co/api-data/api-scrip-master.csv
   - Auto-resolves MCX commodity security IDs (nearest-month FUTCOM)
   - Profile API: GET /v2/profile (auth verification)
   - Option Chain Expiry: POST /v2/optionchain/expirylist
   - Option Chain Data: POST /v2/optionchain
   - Saves data to JSON files for downstream modules
   - Polls dashboard for active instruments list
   - Rate limited: 1 req per 3 sec, 5 sec between full cycles

2. execution_module.py:
   - Auth: Same hardcoded CLIENT_ID + ACCESS_TOKEN
   - Order Placement: POST /v2/orders (MARKET orders, BUY/SELL)
   - Scrip Master: Loads local CSV for security ID lookup
   - find_option_security_id(): Resolves instrument+expiry+strike+type to security_id
   - place_live_order(): Places MARKET orders with correlation ID
   - Paper Trading: Simulated entry/exit using option chain prices
   - Position Monitoring: Reads option chain JSONs, checks SL/TP
   - SL/TP: Monitored in-memory (not Dhan bracket orders)
   - Dashboard Push: POST /api/trading/position, /api/trading/heartbeat
   - Main loop: 5-second cycle

WHAT NEEDS TO CHANGE (Migration to Broker Service)
===================================================

1. Token Management:
   CURRENT: Hardcoded in both Python files
   NEW: MongoDB broker_configs collection, read at startup, auto-refresh popup

2. Order Placement:
   CURRENT: MARKET orders only in execution_module.py
   NEW: LIMIT orders at configurable % below LTP (default 1%)

3. SL/TP:
   CURRENT: Monitored in-memory by execution_module.py (polling option chain)
   NEW: Use Dhan Bracket Orders (BO) so Dhan handles SL/TP triggers server-side

4. Position Tracking:
   CURRENT: In-memory dict in execution_module.py
   NEW: MongoDB, with real-time updates from Dhan WebSocket

5. LTP:
   CURRENT: Read from option chain JSON files (5-sec delay)
   NEW: Dhan WebSocket for real-time tick-by-tick LTP

6. Scrip Master:
   CURRENT: Downloaded in fetcher, loaded from CSV in executor (duplicated)
   NEW: Single download in Broker Service, cached, shared

BROKER SERVICE INTERFACE (Unified API)
======================================

Orders:
  placeOrder(params) → { orderId, status }
  modifyOrder(orderId, params) → { orderId, status }
  cancelOrder(orderId) → { orderId, status }
  exitAll() → { results[] }
  getOrderBook() → orders[]
  getOrderStatus(orderId) → order
  getTradeBook() → trades[]

Positions:
  getPositions() → positions[]
  getHoldings() → holdings[]

Market Data:
  getLTP(instruments) → { instrument: ltp }
  subscribeLTP(instruments, callback) → stream
  getOptionChain(underlying, expiry) → chainData
  getExpiryList(underlying) → dates[]
  getScripMaster(exchange) → instruments[]

Account:
  getMargin() → { available, used, total }
  getFundLimit() → fundDetails
  killSwitch(action) → status

Auth:
  validateToken() → { valid, expiresAt }
  updateToken(newToken) → status

WebSocket Events:
  onOrderUpdate(callback) → stream
  onPositionUpdate(callback) → stream
  onLTPUpdate(instruments, callback) → stream

NO DUPLICATES — CLEAR BOUNDARIES
=================================

Python modules AFTER migration:
  - dhan_option_chain_fetcher.py → Calls Broker Service getOptionChain() instead of Dhan directly
  - execution_module.py → Calls Broker Service placeOrder()/cancelOrder() instead of Dhan directly
  - Both read token from MongoDB (or call Broker Service validateToken())
  - Option chain analysis + AI decision logic stays in Python (no change)

Node.js Broker Service handles:
  - ALL Dhan API communication (single point of contact)
  - Token storage/validation/refresh
  - WebSocket connections (LTP, order updates)
  - Scrip master download and caching
  - Broker adapter abstraction (Dhan now, others later)
